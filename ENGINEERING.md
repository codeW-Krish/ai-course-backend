## 1. SYSTEM OVERVIEW

This backend implements an AI course system that stores course topology in Firestore and generates instructional artifacts only when a learner reaches them. The core path starts in `controller/course.js` with `generateCourseOutline`, which persists a course document and nested `units/subtopics`, then branches into generation endpoints that fill each subtopic’s `content`. Interactive consumption is handled by `controller/interactiveController.js`, where content and quiz retrieval are tied to progression state in `user_subtopic_progress`. Auxiliary artifacts are generated from existing subtopic content by separate controllers: `controller/notesController.js`, `controller/flashcardController.js`, `controller/audioController.js`, and `controller/videoController.js`. Provider abstraction is intentionally thin and implemented in `providers/LLMProviders.js`; transport is plain Express from `index.js` with JWT auth middleware in `middleware/authMiddleware.js`.

The architectural decision that dominates the rest of the implementation is quiz-gated interactive generation, not full eager generation. In practical terms, `getNextContent` in `controller/interactiveController.js` finds the first uncompleted subtopic, generates content if missing, and only then makes that content available. It initializes or reads per-user progression in `user_subtopic_progress`, and progression to the next subtopic is controlled by `submitQuiz`. This changed system behavior in three places. First, content requests now include generation side effects. Second, progress state is first-class and written on almost every interactive call. Third, quiz availability can lag content because quiz generation in `getNextContent` is intentionally fire-and-forget; that introduces eventual consistency between content and assessments, which the client must handle.

The system is built around a three-way tension between API spend, completion behavior, and content quality. Spend pressure is addressed by deferring generation to the point of learner demand and by splitting expensive modalities into narrower stages. Completion pressure is addressed by hard gating in `submitQuiz` and by tracking `hearts_remaining`, `attempts`, and `is_completed`. Quality pressure is addressed by schema validation and structured prompts (`llm/*.js`, `prompts/*.js`) rather than free-form output acceptance. This is visible in repeated `safeParse` checks in `controller/course.js`, `controller/videoController.js`, `controller/notesController.js`, and `controller/audioController.js`, where invalid model output is rejected, skipped, or downgraded to warning paths instead of being silently persisted.

## 2. QUIZ-GATED PROGRESSION — FULL BREAKDOWN

The progression state machine is implemented primarily in `controller/interactiveController.js` through `getNextContent`, `getQuiz`, and `submitQuiz`, with helper calls to `generateContentOnly`, `generateQuizOnly`, `saveQuestions`, and `findSubtopicById`. A subtopic becomes eligible for generation when `getNextContent` scans course subtopics in unit/subtopic order and finds the first record for which `user_subtopic_progress/{userId}_{subtopicId}` does not exist or has `is_completed !== true`. That scan is done by reading `courses/{courseId}/units` ordered by `position`, then each unit’s `subtopics` ordered by `position`, then checking progress docs one by one.

`getNextContent` executes in a fixed sequence. It first materializes ordered subtopics, then selects the first uncompleted candidate. If no candidate exists, it returns `course_completed: true`. If candidate content is missing, it calls `generateContentOnly` and immediately updates the subtopic document with `content`. It then checks `subtopic_questions` for existing quiz records by `subtopic_id`. If no questions exist, it launches an async self-invoked background block that calls `generateQuizOnly` and `saveQuestions`; this block is not awaited. It then ensures a progress document exists for the user/subtopic key with initial fields `{ user_id, subtopic_id, hearts_remaining: 3, attempts: 0, is_completed: false, completed_at: null }`. Finally it computes course progress by re-reading all progress docs and counting completed entries. The response payload includes `subtopic`, `questions_ready`, `hearts_remaining`, `attempts`, `is_completed`, and `course_progress`.

`submitQuiz` writes to more than one collection, but the gating state is stored in `user_subtopic_progress`. It accepts `answers: [{ question_id, user_answer, is_correct }]` and derives `correctCount`, `wrongCount`, `heartsLost`, `heartsRemaining`, `isCompleted`, and `isPerfect`. The write to `user_subtopic_progress/{userId}_{subtopicId}` is a merged set containing `user_id`, `subtopic_id`, `hearts_remaining`, `attempts`, `is_completed`, `completed_at`, and `last_score: { correct, total }`. It also records analytics through `recordQuestionAttempt` in `service/gamificationService.js`, which writes into `user_question_attempts` and daily activity collections. If completed, it awards XP through `grantXpOnce`, which writes an idempotency event into `user_xp_events/{userId}_{eventKey}` and updates aggregate counters in `user_stats`.

Hearts and retries are represented as fields, but there is an implementation detail worth calling out: hearts are recomputed from the current submission’s wrong answers (`3 - min(wrongCount, 3)`) rather than decremented cumulatively from prior attempts. Retries are tracked via `attempts`, which increments each submit, and `last_score`, which stores only the latest submission result. That means retry semantics are “latest attempt snapshot” rather than “lives over session timeline.” Engineers extending mastery logic need to know this because it affects downstream analytics and unlock assumptions.

Partial failure handling between content and quiz is explicit in two places. In `getNextContent`, content generation is synchronous but quiz generation is asynchronous background work; therefore a caller can receive content with `questions_ready: false`. In `getQuiz`, if no quiz docs exist, the handler will attempt to locate the subtopic (`findSubtopicById`), generate missing content if required, then generate and persist questions synchronously. If either generation step fails, `getQuiz` returns a 500 and the client can retry. This two-path design avoids blocking the reading experience on quiz generation while still providing a synchronous fallback endpoint to complete the assessment artifact.

The relevant Firestore collections and relationships are as follows. Course topology lives in `courses/{courseId}/units/{unitId}/subtopics/{subtopicId}`. Quiz questions are denormalized in top-level `subtopic_questions/{subtopicId_position}` with `subtopic_id` as a query key. Per-user progression is `user_subtopic_progress/{userId}_{subtopicId}`. Question-level attempt telemetry is in `user_question_attempts`, and XP idempotency is in `user_xp_events`. The client receives content-first state from `GET /api/interactive/course/:courseId/next-content`, quiz payload from `GET /api/interactive/:subtopicId/quiz`, and progression verdict from `POST /api/interactive/:subtopicId/submit-quiz`.

## 3. EXPLANATION VIDEO PIPELINE — FULL BREAKDOWN

The video pipeline is implemented in `controller/videoController.js` and is intentionally manifest-first. The code does not call a single video-generation model to produce final MP4 output. Instead it generates script and scene plans with an LLM, generates visual assets and narration audio per chunk, and emits a timed manifest that the client plays. The practical reasons visible in code are cost isolation and control isolation. Cost is isolated because script/scene generation uses text LLM calls while visuals and TTS are separately selectable providers (`llm_provider`, `image_provider`, `tts_provider`) and can fall back independently. Control is isolated because each stage has explicit schema checks and deterministic fallbacks (for example, `generateAndUploadAssets` falls back to SVG quote rendering when illustration generation fails). A monolithic video-generation call would collapse these controls.

The sequence is `generateVideoScript` -> `planScenes` -> `generateAndUploadAssets` -> `generateAndUploadAudio` -> `planTransitions` -> `buildManifest`. `generateVideoScript` calls the selected provider with `VIDEO_SCRIPT_PROMPT` from `prompts/videoPrompt.js` and validates with `VideoScriptSchema` from `llm/videoSchemas.js`. `planScenes` calls the same LLM with `SCENE_PLAN_PROMPT` and validates with `ScenePlanListSchema`. Asset generation iterates scenes and either calls `generateIllustration` from `service/imageGenerationService.js` for `illustration` scene types or generates SVG via `generateSVGForScene` from `service/svgGeneratorService.js`, rasterizes with `svgToPngBuffer`, then uploads with `uploadImage` from `service/imagekitService.js`. Audio generation iterates script chunks, synthesizes WAV with `synthesize` from `service/ttsService.js`, measures duration with `getWavDuration`, and uploads with `uploadAudio`.

WAV duration extraction matters because sync is derived from actual audio bytes, not estimated narration length. `getWavDuration` reads RIFF/fmt/data chunks and computes `duration = dataSize / byteRate`; if parsing fails, it falls back to sample-rate assumptions. `buildManifest` converts each chunk duration to milliseconds and uses it to allocate subscene windows (`start_ms`, `duration_ms`). If duration metadata were wrong, scene transition boundaries and text overlays would drift relative to narration. This is why duration is measured after synthesis per chunk rather than trusting `estimated_duration_seconds` from the LLM script output.

`buildManifest` emits a stable object with top-level fields `manifest_version`, `subtopic_id`, `subtopic_title`, `unit_title`, `course_title`, `course_id`, `total_duration_seconds`, `scene_count`, `resolution`, `generated_at`, and `scenes`. Each scene includes `scene_index`, `scene_type`, `visual_url`, `audio_url`, `duration_seconds`, `key_concept`, `narration_text`, `animation_style`, `transition`, and `subscenes`. Each subscene includes `index`, `start_ms`, `duration_ms`, `animation_style`, and `text_overlay`. The manifest is cached in `video_manifests/manifest_{subtopicId}` and served by `getVideoForSubtopic` and `getVideoForCourse`.

The expected client behavior is to fetch the manifest and use scene timing as authoritative playback schedule. The client must preload `visual_url` and `audio_url` assets, schedule scene boundaries from `duration_seconds`, apply transition hints, and render subscene overlays at `start_ms` offsets. The backend does not return rendered MP4 in this path; rendering helpers in `service/videoRenderService.js` exist but are not wired into `videoController.js` request handlers.

Failure and regeneration scope is endpoint-level, not stage-level checkpointing. If a stage throws in `generateSubtopicManifest`, the catch returns status 500 with error text and no partial checkpoint is persisted. Regeneration in `regenerateVideoForSubtopic` deletes only the cached manifest key and re-runs the entire pipeline, which means assets and audio are regenerated too. Provider selection by stage is currently: LLM provider from query for script and scenes (default Groq), image provider cascade in `imageGenerationService.js` (currently `imagegpt` then `deapi`, with `pixazo` commented out in `PROVIDER_CHAIN`), TTS provider from query defaulting to Groq in `ttsService.js`, and transition selection in `service/transitionEngine.js` with no external provider.

## 4. MULTI-PROVIDER LLM ROUTING — FULL BREAKDOWN

`providers/LLMProviders.js` is a registry plus a tiny wrapper, not a policy engine. `LLM_PROVIDERS` maps names (`Gemini`, `Groq`, `Cerebras`, `GLM`) to adapter functions in `service/geminiService.js`, `service/groqService.js`, `service/cerebrasService.js`, and `service/glmService.js`. `getLLMProvider(provider = "Groq", model)` looks up the adapter and returns a closure `(SYSTEM_PROMPT, userInputs) => fn(SYSTEM_PROMPT, { ...userInputs, model })`. This means per-request model override is implemented by adding or replacing the `model` field in the adapter input object.

The default provider from this function is Groq, and most controller entry points also default to Groq when provider is omitted. Examples include `generateCourseOutline` in `controller/course.js`, `getNextContent` and `getQuiz` in `controller/interactiveController.js`, and video generation endpoints in `controller/videoController.js`. The notable exception is `startBackgroundGeneration` in `service/generationQueue.js`, which defaults providerName to `Gemini`; this creates inconsistent defaults between synchronous and queued generation paths.

Routing decisions are made in controller code paths, not centrally. In `controller/course.js`, `generateCourseContent` branches to `generateFullCourseWithCerebras` only when request provider resolves to Cerebras; otherwise it uses `generateWithBatching`. In `streamCourseGeneration`, the same branch is controlled by parsed request body provider. In `controller/videoController.js`, script and scene generation both use whatever `llm_provider` query supplies. In `controller/notesController.js`, `controller/flashcardController.js`, and `controller/audioController.js`, provider and model are query-driven on each endpoint. The routing policy is therefore distributed across handlers and defaults, which is a known governance limitation because behavior is hard to reason about globally and drift can occur.

Model overrides per request are straightforward but permissive. Controllers pass provider and model from `req.body` or `req.query` into `getLLMProvider`; adapters then choose `userInputs.model || defaultModel`. This allows runtime experimentation without deployment changes, but there is no central allowlist or model capability check. A malformed model value will fail at provider call time rather than at API validation time.

Validation uses Zod at multiple points. In `llm/outlineSchemas.js`, `OutlineRequestSchema` validates user input, `LlmOutlineSchema` validates normalized outline output, `RegenerateContentOutlineSchema` validates regeneration payload, and `SubtopicBatchResponseSchema` validates per-subtopic generated content including required `why_this_matters`. In `llm/interactiveSchemas.js`, `InteractiveContentSchema`, `ContentOnlySchema`, and `QuizOnlySchema` validate interactive outputs. In `llm/videoSchemas.js`, script and scene structures are validated before manifest assembly. Rejection behavior varies by endpoint: some return immediate 400 or 502 (`generateCourseOutline`, `generateFullCourseWithCerebras`), while streaming and batched paths can downgrade to warning events and continue (`processStreamingBatches` in `controller/course.js`).

## 5. SSE STREAMING ARCHITECTURE — FULL BREAKDOWN

SSE generation is implemented by `streamCourseGeneration` in `controller/course.js` and exposed by `POST /api/courses/:id/generate-content-stream` in `routes/course.js`. The handler writes SSE headers (`text/event-stream`, `no-cache`, `X-Accel-Buffering: no`) and defines a `send` helper that emits JSON frames as `data: ...\n\n`. It manually reads request body chunks and parses provider/model because JSON body parsing is intentionally bypassed for this endpoint in `index.js`.

Event types are explicit and finite. `type: "start"` is emitted once after missing subtopics are collected, including `total`, `provider`, and `isCerebras`. `type: "chunk"` is emitted in non-Cerebras batch mode after a batch validates, with a pretty-printed serialized chunk payload. `type: "progress"` is emitted whenever one subtopic is successfully persisted to Firestore, carrying `subtopic`, `unit`, percentage progress, generated count, and total. `type: "warning"` is emitted for recoverable per-batch problems (invalid response, schema failure, match miss, or batch exception in streaming mode). `type: "complete"` is emitted at end with generated and total counts. `type: "error"` is emitted on fatal request errors (invalid body, ownership failure, top-level generation failure).

Cerebras and non-Cerebras differ in granularity. `processCerebrasBatch` builds one course-wide input with all missing subtopics grouped by unit, expects one large response, validates once, and then persists item by item while emitting `progress`. `processStreamingBatches` chunks each unit into batches of three subtopics, validates each batch independently, emits `chunk` and `progress` per successful item, and emits `warning` for local failures while continuing with later batches. This makes non-Cerebras mode more resilient to partial failures at the cost of more requests.

Progress signaling during generation is realtime via `progress` events, while persistence-level status for polling is handled by `course_generation_status` writes. `streamCourseGeneration` writes final status once at completion. `startBackgroundGeneration` in `service/generationQueue.js` writes incremental `generated_subtopics` and terminal `status`. The polling endpoint `getCourseGenerationStatus` can return generated subtopics and supports `since` filtering. This split means clients can combine SSE for live UX and polling for post-disconnect reconciliation.

Reconnect and partial state recovery are mostly client responsibilities. The server does not emit SSE `id` fields, does not process `Last-Event-ID`, and does not support resumable stream offsets. What exists is external recovery via Firestore-backed status and already-persisted subtopic content. Known failure modes include invalid JSON request body on SSE endpoint, provider/schema mismatch in a batch, write failures while persisting content, and stream interruption by network/proxy. The current handling is either warning-and-continue for batch-local failures or terminal error for top-level exceptions.

## 6. CONTENT GENERATION PIPELINE — FULL BREAKDOWN

Outline generation starts in `generateCourseOutline` (`controller/course.js`) with request normalization and validation via `OutlineRequestSchema`. It calls the selected provider with `OUTLINE_SYSTEM_PROMPT` (`prompts/outlinePrompt.js`), normalizes output through `normalizeLlmOutline`, and validates using `LlmOutlineSchema`. Persistence then creates `courses/{courseId}` with metadata (`created_by`, `title`, `description`, `difficulty`, `include_videos`, `status`, `outline_json`, timestamps) and creates ordered nested `units` and `subtopics` where each subtopic starts with `content: null` and `content_generated_at: null`. This staged persistence is what allows later generation strategies to target only missing subtopics.

Two generation strategies exist and are selected by provider branch. `generateCourseContent` dispatches to `generateFullCourseWithCerebras` only when provider name is Cerebras; otherwise it uses `generateWithBatching`. `generateWithBatching` currently limits to the first unit in synchronous API mode (`orderBy(position).limit(1)`), identifies missing subtopics, chunks them (single batch if <=4 else groups of 3), calls `SUBTOPIC_BATCH_PROMPT`, validates each response against `SubtopicBatchResponseSchema`, and writes content into each subtopic document. `generateFullCourseWithCerebras` collects all missing subtopics across all units, sends one full input, validates once, and persists matched subtopics across the course.

The per-subtopic content structure enforced by `SubtopicContentSchema` in `llm/outlineSchemas.js` is: `subtopic_title`, `title`, `why_this_matters`, `core_concepts` (array of `{ concept, explanation }`), `examples` (array of typed example objects), `code_or_math` (string or null), and optional `youtube_keywords` (array or null). `SUBTOPIC_BATCH_PROMPT` in `prompts/SubTopicBatchPrompt.js` enforces exact subtopic title echoing and explicit nulls where needed. Match logic in persistence is title-based normalization (`toLowerCase`, whitespace collapse), which is convenient but creates potential collision risk if different units contain same normalized subtopic titles.

Parallel feature generators are decoupled and on-demand rather than fan-out from main content generation. Notes generation in `controller/notesController.js` reads subtopic content and stores `generated_notes/{subtopicId}`. Flashcards generation in `controller/flashcardController.js` writes to top-level `flashcards` with per-card progress in `user_flashcard_progress`. Audio generation in `controller/audioController.js` writes cached docs in `generated_audio/sub_{subtopicId}` and `generated_audio/course_{courseId}`, with stream/download endpoints that can regenerate if the cached CDN URL is stale. None of these are triggered automatically by `generateWithBatching` or `generateFullCourseWithCerebras`; they are endpoint-triggered by client demand.

`why_this_matters` is defined in both the prompt contract and schema contract. It is explicitly requested in `SUBTOPIC_BATCH_PROMPT` and required by `SubtopicContentSchema` (`z.string()`). If missing, `SubtopicBatchResponseSchema.safeParse` fails. The failure consequence depends on context: synchronous endpoints may skip a batch item or return an error; SSE streaming emits warnings and continues processing later batches. This is a deliberate strictness choice to keep pedagogical structure enforced at storage boundaries.

## 7. KNOWN LIMITATIONS AND WHAT I WOULD CHANGE

Routing policy governance is currently scattered. The current approach puts provider defaults and branching rules inside controllers (`controller/course.js`, `controller/interactiveController.js`, `controller/videoController.js`) and in one queue service (`service/generationQueue.js`) with conflicting default provider behavior. It was likely built this way for speed and endpoint autonomy during feature expansion. The better approach is a central routing module that accepts task metadata (`taskType`, latency target, cost target, reliability target) and returns provider/model decisions plus fallback policy. Changing this requires extracting all `provider || "Groq"` and Cerebras branch checks into a single policy layer, adding an allowlist for models, and instrumenting routing outcomes.

Environment management is incomplete. There is a real `.env` ignored by `.gitignore`, but no committed `.env.example` template and no machine-readable config contract. This is common in early delivery phases but creates onboarding friction and hidden runtime assumptions. The better approach is to commit `.env.example` with all required keys grouped by subsystem (auth, Firebase, LLM, image, TTS, YouTube, optional render tools), plus startup validation that fails fast with actionable errors. Implementing this requires a small config module, schema validation at boot, and docs aligned with `package.json` scripts.

Retry and idempotency are uneven in generation paths. Some calls have retries (`withRetry` in `interactiveController.js`), while others do not (`generateWithBatching`, most of `videoController.js`). Idempotency keys are present for XP events (`grantXpOnce`) but not for subtopic content generation jobs. This was likely acceptable during initial throughput tuning. A better design would add job records with deterministic job IDs per `(courseId, subtopicId, artifactType, provider, model)` and exactly-once persistence semantics using transactions or compare-and-set guards. The change requires adding a `generation_jobs` collection, wrapping writes in transaction checks, and introducing bounded retry with backoff and dead-letter state.

Subtopic lookup patterns are expensive in multiple controllers. `findSubtopicWithContext` in notes, flashcards, audio, and video controllers scans all courses and units to locate one subtopic ID. This likely came from avoiding additional indexes and keeping document paths opaque to clients. A better approach is to persist reverse index records (for example `subtopic_index/{subtopicId}` -> `{ courseId, unitId }`) when outlines are created or modified. Migration requires backfilling existing subtopics and updating creation/regeneration code paths to maintain index consistency.

SSE recovery semantics are limited. The stream endpoint has no event IDs, no resume tokens, and no reconnect-aware replay, so transient disconnects lose in-flight telemetry. The fallback is polling `getCourseGenerationStatus`, which is useful but not equivalent to replayable stream state. This was likely acceptable for a first streaming iteration. The better approach is to persist event checkpoints with monotonic sequence numbers and support `Last-Event-ID` for replay windows. Implementing this requires a lightweight event log per job and retention cleanup logic.

There is also a data-shape inconsistency around provider settings in admin routes. `routes/admin.js` writes `available_providers` as `JSON.stringify(providers)`, but `models/GlobalSettings.js` expects raw values and returns `doc.data().value` directly. Depending on how values are written elsewhere, this can produce string-vs-array behavior and break `.includes` checks in provider validation. The likely reason is incremental admin tooling changes without migration. The better approach is to enforce typed settings in one schema and migrate existing docs to canonical types.

Finally, there are stale or misleading code paths that increase maintenance cost. `service/videoRenderService.js` provides MP4 rendering logic but is not called from `controller/videoController.js`, which returns manifests only. `prompts/subTopicSystemPrompt.js` exists but the active batch generation path uses `prompts/SubTopicBatchPrompt.js`. These leftovers are common when architecture pivots, but they blur current truth for new contributors. The better approach is either to remove unused modules or mark them as experimental with explicit ownership and activation criteria.
